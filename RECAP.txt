1. STRUCTURE DU PROJET
-----------------------------------------------------------------------
  bootstrap.sh                  Script principal (18 étapes idempotentes)
  lib/gen_microservices.sh      Génère API Express + Front nginx
  lib/gen_manifests.sh          Génère manifests Kubernetes
  lib/gen_skills.sh             Génère 4 Skills Claude Code
  lib/gen_workflow.sh           Génère workflow GitHub Actions
  lib/prepare_server.sh         Provisionne le serveur distant
  tp-devops-agent-ia/           Dossier de sortie (microservices + k8s)
  .bootstrap-state              Suivi d'avancement (idempotence)
  .bootstrap-env                Credentials (chmod 600, non versionné)


2. PORTS UTILISÉS
-----------------------------------------------------------------------
  Ports applicatifs (configurables au prompt) :
    APP_PORT   80     Frontend nginx (web)        [conteneur + Service]
    API_PORT   3000   API Express (backend)       [conteneur + Service]

  Ports Kubernetes Services (toujours 80 côté Service) :
    Service api    80 -> targetPort API_PORT (3000)
    Service web    80 -> targetPort APP_PORT (80)

  Ports exposés / pare-feu (UFW, lib/prepare_server.sh) :
    22/tcp     SSH
    80/tcp     HTTP (Ingress / Traefik)
    443/tcp    HTTPS (Ingress / Traefik)
    6443/tcp   API Kubernetes (kube-apiserver k3s)

  Endpoints exposés en sortie :
    http://OVH_HOST/            Frontend
    http://OVH_HOST/api/        API
    http://OVH_HOST/api/health  Health-check


3. CLUSTERS
-----------------------------------------------------------------------
  Distribution     : k3s (lightweight Kubernetes)
  Topologie        : mono-nœud (serveur OVH cible unique)
  Installation     : curl -sfL https://get.k3s.io | sh -
                     (avec --write-kubeconfig-mode 644)
  Ingress          : Traefik (intégré k3s par défaut)
  Metrics          : metrics-server (avec --kubelet-insecure-tls)
  Namespace utilisé: default (aucun namespace explicite dans les manifests)
  Kubeconfig local : ~/.kube/config-<GITHUB_REPO_NAME>
                     (127.0.0.1 réécrit en OVH_HOST)

  Objets déployés dans le cluster :
    Deployment  <APP_NAME>-api     (REPLICAS_API réplicas)
    Deployment  <APP_NAME>-web     (REPLICAS_WEB réplicas)
    Service     <APP_NAME>-api     (ClusterIP)
    Service     <APP_NAME>-web     (ClusterIP)
    Middleware  strip-api-prefix   (Traefik)
    Ingress     <APP_NAME>         (routes / et /api)


4. PIPELINE / CI-CD
-----------------------------------------------------------------------
  Runner       : GitHub Actions (ubuntu-latest, AMD64)
  Build        : Docker images (api + web) -> Docker Hub
  Déploiement  : SSH -> kubectl apply -f k8s/base/
  Secrets GH   : DOCKERHUB_USERNAME, DOCKERHUB_TOKEN,
                 OVH_HOST, OVH_USER, OVH_SSH_KEY


5. CORRECTIONS APPLIQUÉES AU SCRIPT
-----------------------------------------------------------------------
  - bootstrap.sh ligne 737 : suppression du flag obsolète `--confirm`
    pour `gh repo create` (deprecated dans gh CLI ≥ 2.x, provoquait un
    `unknown flag` puis fallback inutile).


6. POINTS D'ATTENTION (non bloquants)
-----------------------------------------------------------------------
  - Les manifests utilisent le namespace `default` ; envisager un
    namespace dédié (ex: tp-app-<DEPLOY_ENV>) pour isoler dev/staging/prod.
  - Aucun NetworkPolicy ni PodSecurity n'est défini.
  - Le port 6443 est ouvert publiquement (UFW) : à restreindre par IP
    en production.
  - Le kubeconfig contient un token cluster-admin : protéger ~/.kube/.
